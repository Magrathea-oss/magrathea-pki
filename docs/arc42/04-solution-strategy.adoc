= 4. Solution Strategy

Status: planned with confirmed architectural direction; implementation is still early.

== 4.1 Strategic Drivers

The solution strategy follows the Chapter 1 priority order:

. Security
. Correctness
. Auditability
. Modularity
. Testability
. Operability

== 4.2 Domain-Centric Architecture

Status: confirmed.

Magrathea PKI uses DDD to keep protocol and custody concerns explicit:

* X.509 CA, CVC CA, and SSH CA remain separate bounded contexts.
* KMS handles key lifecycle, cryptographic operations, policy checks, and custody backend selection.
* HSM Key Custody and TPM Attestation are optional trust/key-custody profiles inside the solution boundary when product-owned or product-managed.
* Shared kernels are allowed only for truly shared concepts and must not collapse protocol-specific semantics.

== 4.3 Roadmap Strategy

Status: confirmed.

The product roadmap is:

. X.509 first;
. CVC second;
. SSH third.

Current executable requirements cover X.509 profile lifecycle event-envelope metadata, X.509 profile integrity, and KMS; all observed scenarios remain tagged `@not-implemented`. After the dataset reset, X.509 implementation evidence is limited to replacement episode 0001: `CertificateProfileId` preserves and exposes one supplied string value. The normative CertificateProfile v1 identity strategy is confirmed by link:../adr/0001-use-content-addressed-yaml-for-certificate-profiles.md[ADR 0001], but grammar enforcement, the six-field semantic model, strict YAML/schema admission, default expansion, NFC and unordered-set normalization, RFC 8785 JCS serialization, SHA-256 derivation, trusted-manifest comparison, audit emission, lifecycle envelopes, and application behavior are not implemented. Broader X.509, CVC, and SSH feature coverage is still missing.

== 4.4 Technology Strategy

[cols="1,1,3",options="header"]
|===
| Area | Status | Strategy

| Runtime
| confirmed
| Java 21.

| Application platform
| confirmed/implementation-informed
| Spring Boot 4.1.0 with Spring Reactive/WebFlux on Java 21. The scaffold has been converted from Spring Boot 3.4.4; Reactor and R2DBC dependency versions are managed by Spring Boot.

| Reactive design
| planned
| Application ports and adapters should support reactive flows where required by WebFlux and operational scalability.

| Certificate profile registry/lifecycle source
| confirmed design / not implemented
| link:../adr/0002-use-git-backed-governed-event-store-for-certificate-profile-lifecycle.md[ADR 0002] chooses a Git-backed governed event store as the low-throughput source for certificate profile registry/lifecycle events and immutable profile artifact/manifest publication. CQRS keeps reads replaceable: simple Git-derived reads/caches may be used first, while PostgreSQL, OpenSearch, cache projections, or a dedicated event store remain available later. `features/x509/certificate-profile-lifecycle.feature` records the event-envelope requirement. No lifecycle-envelope domain implementation, Git event-store adapter, projectors, read models, CAS, signed commits, replay, or runtime support exists yet.

| Build/scaffold
| implementation-informed
| Existing Maven modules express domain, application, infrastructure, API adapter, CLI adapter, and bootstrap layers, but do not define C4 containers. Full reactor packaging has passed with Spring Boot 4.1.0 and Java 21.
|===

== 4.5 Quality-Driven Strategy

[cols="1,3",options="header"]
|===
| Quality | Architectural Response

| Security
| Keep key custody explicit; isolate HSM/TPM/software custody choices; enforce policy before cryptographic use.

| Correctness
| Preserve protocol-specific bounded contexts and executable requirements.

| Auditability
| Treat audit evidence as a cross-cutting concern and a shared kernel candidate.

| Modularity
| Separate bounded contexts from Maven modules and packaging artifacts.

| Testability
| Use Gherkin as executable requirements and javaspec for focused Java behavior slices.

| Operability
| Keep deployment placement and runtime observability explicit in later deployment decisions.
|===

== 4.6 Architecture Decisions

Status: implementation-informed.

Existing strategy decisions are indexed in chapter 9, including link:../adr/0001-use-content-addressed-yaml-for-certificate-profiles.md[ADR 0001] for content-addressed YAML certificate profiles and link:../adr/0002-use-git-backed-governed-event-store-for-certificate-profile-lifecycle.md[ADR 0002] for the Git-backed governed profile lifecycle event-store direction.

Major strategy decisions still need ADRs, especially for:

* bounded-context boundaries;
* Spring Boot 4 LTS platform baseline and dependency-management policy;
* key custody profiles and cryptographic boundary rules;
* broader audit storage and immutability beyond the governed profile lifecycle source;
* compliance evidence approach.
