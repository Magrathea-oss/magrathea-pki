= 11. Risks and Technical Debt

|===
| Item | Status | Impact | Mitigation / Next Action | Owner

| Missing `trust-engine-admin-adapter` and `trust-engine-test-support`
| debt
| Repository policy describes modules that the scaffold does not yet contain.
| Decide whether to add modules or update policy; do not assume they exist.
| bdd-java-scaffolder

| Partial X.509 feature coverage
| risk
| X.509 now has profile-integrity requirements `001`–`014`, one lifecycle-envelope requirement, subject-public-key-profile requirements `001`–`005`, and deployment-policy requirements `001`–`016`. Complete behavior scenarios are `@not-implemented`, while deployment-policy `013`–`016` are unresolved `@placeholder` decisions. Profile management beyond the envelope, issuance, revocation, and validation scenarios remain missing.
| Implement one coherent requirement slice at a time and resolve placeholders through human/ADR review; retain anti-fake status until semantic validation passes.
| bdd-java-planner / spec-driven / story-driven / product and security owners

| CertificateProfile v1 contract not implemented
| risk / confirmed design / partial domain evidence
| link:../adr/0001-use-content-addressed-yaml-for-certificate-profiles.md[ADR 0001] fixes the six-field schema, strict admission, normalization, JCS bytes, SHA-256 identity, ID grammar, and identity/issuance-policy boundary. Episodes 0001–0003 prove canonical identifier preservation, leading-whitespace rejection, and explicit-null rejection only; they do not complete `REQ-X509-PROFILE-INTEGRITY-014` or the profile pipeline. The discarded 33-behavior corpus is not support evidence.
| Complete the remaining grammar/reader examples and implement the normative pipeline incrementally against requirements `001` through `014`. Do not claim parsing, canonicalization, hashing, manifest comparison, audit emission, or issuance eligibility before corresponding validation is GREEN.
| security architect / spec-driven / story-driven

| Manifest signing and trusted distribution unresolved
| risk
| A content hash detects tampering only when compared with a separately trusted expected ID. An attacker able to alter both profile YAML and its expected reference can bypass the comparison.
| Decide the first-release trust/distribution model and the trigger for signed manifests, software or HSM signatures, signed Git publication, append-only audit, or transparency mechanisms.
| security architect / compliance officer / operations

| Identity could be confused with issuance safety
| risk
| A stable matching profile ID does not validate subject-template semantics, cross-field constraints, lifecycle eligibility, authorization, or other issuance policy. Treating identity success as approval could enable unsafe issuance.
| Keep identity formation and issuance-policy validation as separate fail-closed gates, as required by ADR 0001. Define and approve issuance rules before implementing certificate issuance.
| PKI architect / security architect / story-driven

| X.509 profile lifecycle edge cases unresolved
| risk
| Ambiguous transitions among `draft`, `active`, `deprecated`, `revoked`, and `archived` could cause unsafe issuance or unclear historical traceability.
| Define transition, alias movement, rollback, and issuance-eligibility rules.
| security architect / product owner / story-driven

| Git-backed profile lifecycle event store not implemented
| risk / confirmed design
| link:../adr/0002-use-git-backed-governed-event-store-for-certificate-profile-lifecycle.md[ADR 0002] chooses a governed low-throughput Git source, but no lifecycle-envelope domain behavior, Git adapter, protected stream layout, signed publication, expected-head/CAS path, projector, read model, replay validation, or runtime integration exists.
| Complete the lifecycle envelope requirement and define stream layout, trust controls, event schema, and projection/replay validation before claiming support.
| security architect / story-driven / adr-writer

| All observed KMS scenarios are `@not-implemented`
| risk
| Documentation could overstate current product support.
| Keep implementation status explicit until semantic validation passes.
| story-driven / spec-driven

| Algorithm taxonomy implementation lags ADR 0003
| risk / partial domain evidence
| Episode 0004 proves only the bare exact `EC_P256`/`RSA_3072` vocabulary. Depending on Java enum names, mixing protocol roles, or adding deployment assertions to enums would violate stable-wire and boundary rules; the rename could accidentally alter CertificateProfile v1 identity.
| Implement explicit wire tokens, exact parsing/serialization, role-specific taxonomies, mutable-assertion exclusion, and golden-byte rename compatibility against requirements `001`–`005`.
| PKI architect / spec-driven / story-driven

| Deployment-policy trust and runtime model not implemented
| high-impact risk / confirmed design
| ADR 0004 requires strict admission, canonical identity, signatures/trust anchors, immutable evidence, deterministic explicit-time evaluation, fail-closed decisions, rollback state, and atomic reload. None exists; `REQ-X509-DEPLOYMENT-POLICY-001`–`012` are `@not-implemented`, and `013`–`016` leave critical governance choices unresolved.
| Resolve placeholders before implementation, threat-model publication/evaluator supply chains, and deliver requirement slices without treating Git or content hashes as trust/freshness proofs.
| security architect / operations / compliance officer / product owner

| JavaSpec RC1 modeling limitation
| technical risk
| JavaSpec 1.0.0-RC1's observed generator cannot coherently generate heterogeneous top-level sealed permittees. Tooling pressure could distort the production model or lead to false completion.
| Use enriched enums or narrow structured types where architecturally appropriate; if a justified sealed model hits the limitation, stop with `FRAMEWORK_INCOHERENCE` assigned to JavaSpec rather than work around it in production design.
| spec-driven / JavaSpec owner

| Dataset evidence could be overstated as RL-ready
| governance risk
| Episodes 0001–0004 are accepted raw records and curated SFT candidates, but evaluation eligibility is `NOT_ELIGIBLE`. Passing behavior/replay gates do not provide an RL reward contract, hidden evaluation, partition controls, state/action evidence, or preference alternatives.
| Preserve independent dataset ownership and make no RL-readiness claim until those gates are separately defined and approved.
| dataset-writer / evaluation owner

| Broad compliance targets without scoped certification strategy
| risk
| FIPS, eIDAS, ETSI, and Common Criteria can impose conflicting or expensive obligations. Algorithm tokens and policy decisions do not establish certification or general conformance.
| Define edition- and deployment-scoped compliance goals, evidence issuers/retention, concrete evaluated configurations, and target assurance levels.
| product owner / security architect / compliance officer

| Key custody profile details unresolved
| risk
| Software, HSM, and TPM profiles affect security boundaries, deployment, and testing.
| Write ADRs for custody boundaries and supported provider profiles.
| security architect / adr-writer

| Audit immutability design not specified
| risk
| Auditability is a priority but storage, retention, export, and tamper evidence are undefined outside the bounded profile-lifecycle decision.
| Define the broader audit store, immutability mechanism, retention, and evidence export requirements.
| security architect / compliance officer

| ADR coverage incomplete
| debt
| Algorithm taxonomy and deployment-policy boundaries now have ADRs 0003/0004, but platform, bounded-context, custody, broader audit, and concrete compliance decisions remain incomplete. ADR 0004 also requires follow-up identity, trust, recovery, evidence, and catalog decisions.
| Continue the ADR set without duplicating ADR bodies in ARC42.
| adr-writer / arc42

| No deployment view
| gap
| HSM/TPM placement, runtime topology, and operator access boundaries remain unclear.
| Confirm deployment profiles and ask c4model to add a deployment view.
| c4model / operations

| C4 model ahead of implementation
| risk
| C4 containers and KMS components are planned design elements, not confirmed production implementations.
| Keep status markers and update C4 after implementation slices.
| c4model / arc42

| Minimal production implementation
| debt
| Beyond the bootstrap scaffold, accepted X.509 domain evidence is limited to episodes 0001–0003 for `CertificateProfileId` preservation/grammar/null behavior and episode 0004 for the exact bare `PublicKeyAlgorithm` two-token vocabulary. Enrichment, rename, policy YAML/evaluator, and every complete feature scenario remain unimplemented.
| Continue requirement-traced RED/GREEN slices before claiming broader production behavior.
| spec-driven / story-driven
