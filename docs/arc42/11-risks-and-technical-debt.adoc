= 11. Risks and Technical Debt

Status: implementation-informed and needs-review.

[cols="1,1,2,2,1",options="header"]
|===
| Item | Status | Impact | Mitigation / Next Action | Owner

| Missing `trust-engine-admin-adapter` and `trust-engine-test-support`
| debt
| Repository policy describes modules that the scaffold does not yet contain.
| Decide whether to add modules or update policy; do not assume they exist.
| bdd-java-scaffolder

| Partial X.509 feature coverage
| risk
| Profile integrity now has requirements `REQ-X509-PROFILE-INTEGRITY-001` through `014`, and profile lifecycle has one event-envelope requirement. Profile management beyond that envelope, issuance, revocation, and validation scenarios remain missing. Every observed X.509 scenario is `@not-implemented`.
| Implement one requirement slice at a time through the BDD-first workflow and retain anti-fake status until semantic validation passes.
| bdd-java-planner / spec-driven / story-driven

| CertificateProfile v1 contract not implemented
| risk / confirmed design
| link:../adr/0001-use-content-addressed-yaml-for-certificate-profiles.md[ADR 0001] now fixes the six-field schema, strict admission boundary, normalization, JCS bytes, SHA-256 identity, ID grammar, and identity/issuance-policy boundary. Implementation drift could create incompatible IDs or unsafe acceptance. Replacement episode 0001 proves only that `CertificateProfileId` preserves and exposes a supplied string; it does not validate the v1 grammar. The historical 33-behavior corpus was discarded and is not support evidence.
| Implement and validate the normative pipeline incrementally against requirements `001` through `014`, including the golden vector. Do not claim parsing, canonicalization, hashing, manifest comparison, audit emission, or issuance eligibility before corresponding validation is GREEN.
| security architect / spec-driven / story-driven

| Manifest signing and trusted distribution unresolved
| risk
| A content hash detects tampering only when compared with a separately trusted expected ID. An attacker able to alter both profile YAML and its expected reference can bypass the comparison.
| Decide the first-release trust/distribution model and the trigger for signed manifests, software or HSM signatures, signed Git publication, append-only audit, or transparency mechanisms.
| security architect / compliance officer / operations

| Identity could be confused with issuance safety
| risk
| A stable matching profile ID does not validate subject-template semantics, cross-field constraints, lifecycle eligibility, authorization, or other issuance policy. Treating identity success as approval could enable unsafe issuance.
| Keep identity formation and issuance-policy validation as separate fail-closed gates, as required by ADR 0001. Define and approve issuance rules before implementing certificate issuance.
| PKI architect / security architect / story-driven

| X.509 profile lifecycle edge cases unresolved
| risk
| Ambiguous transitions among `draft`, `active`, `deprecated`, `revoked`, and `archived` could cause unsafe issuance or unclear historical traceability.
| Define transition, alias movement, rollback, and issuance-eligibility rules.
| security architect / product owner / story-driven

| Git-backed profile lifecycle event store not implemented
| risk / confirmed design
| link:../adr/0002-use-git-backed-governed-event-store-for-certificate-profile-lifecycle.md[ADR 0002] chooses a governed low-throughput Git source, but no lifecycle-envelope domain behavior, Git adapter, protected stream layout, signed publication, expected-head/CAS path, projector, read model, replay validation, or runtime integration exists.
| Complete the lifecycle envelope requirement and define stream layout, trust controls, event schema, and projection/replay validation before claiming support.
| security architect / story-driven / adr-writer

| All observed KMS scenarios are `@not-implemented`
| risk
| Documentation could overstate current product support.
| Keep implementation status explicit until semantic validation passes.
| story-driven / spec-driven

| Broad compliance targets without scoped certification strategy
| risk
| FIPS, eIDAS, ETSI, and Common Criteria can impose conflicting or expensive obligations.
| Define compliance scope, evidence model, and target assurance levels.
| product owner / security architect / compliance officer

| Key custody profile details unresolved
| risk
| Software, HSM, and TPM profiles affect security boundaries, deployment, and testing.
| Write ADRs for custody boundaries and supported provider profiles.
| security architect / adr-writer

| Audit immutability design not specified
| risk
| Auditability is a priority but storage, retention, export, and tamper evidence are undefined outside the bounded profile-lifecycle decision.
| Define the broader audit store, immutability mechanism, retention, and evidence export requirements.
| security architect / compliance officer

| ADR coverage incomplete
| debt
| Major platform, bounded-context, custody, policy, audit, and compliance decisions still lack durable records.
| Continue the ADR set without duplicating ADR bodies in ARC42.
| adr-writer / arc42

| No deployment view
| gap
| HSM/TPM placement, runtime topology, and operator access boundaries remain unclear.
| Confirm deployment profiles and ask c4model to add a deployment view.
| c4model / operations

| C4 model ahead of implementation
| risk
| C4 containers and KMS components are planned design elements, not confirmed production implementations.
| Keep status markers and update C4 after implementation slices.
| c4model / arc42

| Minimal production implementation
| debt
| Beyond the bootstrap scaffold, current accepted X.509 domain evidence is only `CertificateProfileId` value preservation from replacement episode 0001. No feature scenario is implemented.
| Continue requirement-traced RED/GREEN slices before broader production behavior.
| spec-driven / story-driven
