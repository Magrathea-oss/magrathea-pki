= 8. Cross-cutting Concepts

Status: planned with limited implementation evidence.

== 8.1 Security and Key Custody

Status: confirmed/planned.

Magrathea PKI must protect keys, trust decisions, certificate lifecycle operations, and administrative access. Supported custody profiles are:

* software-based key custody;
* optional HSM-backed custody;
* optional TPM-backed trust and attestation.

HSM and TPM capabilities are inside the solution boundary when product-owned or product-managed. Concrete cryptographic provider boundaries, key export rules, and custody failure modes need ADRs.

== 8.2 Policy Enforcement

Status: planned and feature-informed.

KMS feature files require policy checks for:

* allowed usages such as `SIGN` and `ENCRYPT_DECRYPT`;
* required backend type such as `HSM` or `SOFTWARE`;
* required approval before use;
* audit events for policy-denied operations.

== 8.3 Auditability

Status: planned and feature-informed.

KMS feature files require audit events for key creation, signing, destruction, policy denial, and audit immutability violations. Audit events must be append-only and include key handle, operation type, timestamp, and identity where applicable.

Audit storage, retention, export, tamper evidence, and compliance evidence packaging are not yet specified.

== 8.4 Validation and Error Handling

Status: planned.

Errors must preserve security and correctness. Requirement-informed examples include:

* policy denial for disallowed usage;
* policy denial for unavailable required backend;
* approval-required denial;
* decryption failure with the wrong key;
* invalid verification result with the wrong signing key.

Concrete error contracts are not yet defined.

== 8.5 Persistence and Transaction Boundaries

Status: needs-review.

The C4 KMS component view includes a Key Repository design element, but persistence technology, transaction boundaries, and audit immutability storage are not yet decided.

== 8.6 Observability and Operability

Status: planned.

The architecture needs diagnostics and operational visibility, especially for custody backend health, policy denials, audit failures, and CA lifecycle operations. Metrics, tracing, log structure, and alerting targets are not yet specified.

== 8.7 Configuration and Secrets

Status: needs-review.

Secrets, cryptographic provider configuration, HSM credentials, TPM provisioning data, and policy configuration require explicit handling. No concrete mechanism is documented yet.

== 8.8 Configuration Integrity and Tamper-evident Profile Storage

Status: confirmed decision / not implemented, with one implementation-informed value-preservation slice.

link:../adr/0001-use-content-addressed-yaml-for-certificate-profiles.md[ADR 0001] is normative for the `x509-profile-v1-sha256` identity namespace. X.509 certificate profiles are immutable, content-addressed YAML resources, and identity is derived from the admitted semantic model rather than raw YAML bytes.

The cross-cutting identity mechanism is:

[source,text]
----
YAML parse -> strict schema admission -> six-field model/default expansion
-> subjectTemplate NFC + unordered-set sorting -> RFC 8785 JCS UTF-8 bytes
-> SHA-256 -> namespaced profile ID -> trusted-reference comparison
----

At this abstraction level, implementations must preserve these properties:

* after defaults, the model contains exactly `subjectTemplate`, `publicKeyAlgorithms`, `validityDays`, `keyUsages`, `extendedKeyUsages`, and `basicConstraints`;
* admission is fail-closed for malformed or ambiguous YAML, wrong native types, unknown fields or enum values, duplicate keys or set members, explicit nulls, and YAML composition features forbidden by ADR 0001;
* NFC applies only to `subjectTemplate`; its other accepted characters remain identity-significant;
* only the three declared enum collections are unordered sets, validated before unsigned UTF-8 lexical sorting;
* RFC 8785 JCS defines canonical JSON serialization, and SHA-256 hashes exactly its UTF-8 bytes without a BOM or trailing newline;
* a v1 identifier is a whole-string `x509-profile-v1-sha256-` prefix followed by 64 lowercase hexadecimal characters, without trimming;
* identity-affecting changes require a new namespace, while existing IDs remain immutable.

The detailed field limits, closed vocabularies, defaults, rejection rules, and golden fixture are intentionally not duplicated here; they remain governed by ADR 0001 and executable requirements `REQ-X509-PROFILE-INTEGRITY-001` through `014`.

Expected IDs are stored separately in `profiles/profiles.manifest.yaml` or another trusted reference. Before issuance can proceed to policy validation, computed and expected IDs must match. A mismatch rejects issuance eligibility and requires `PROFILE_INTEGRITY_FAILED` evidence containing the profile path, expected and computed IDs, actor or system identity, timestamp, and reason. Identity success alone never establishes issuance safety: subject-template semantics, cross-field rules, lifecycle eligibility, authorization, and other issuance policy remain a separate fail-closed gate.

Lifecycle states are `draft`, `active`, `deprecated`, `revoked`, and `archived`. The first release does not require manifest signatures, but the trust/distribution model remains an open security decision.

Anti-fake status: accepted replacement episodes 0001–0003 prove only that the current `CertificateProfileId` record preserves an admitted canonical value and rejects leading whitespace and explicit null under the complete v1 grammar. This is implementation evidence for domain invariants, not complete support for `REQ-X509-PROFILE-INTEGRITY-014`, whose scenario outline includes additional malformed values and reader behavior. No CertificateProfile model, YAML admission, defaulting, normalization, JCS serialization, hashing, manifest comparison, audit emission, or application/adapter behavior is implemented; all profile-integrity scenarios remain `@not-implemented`.

== 8.9 Governed Profile Lifecycle Event Store and CQRS Projections

Status: confirmed design / not implemented.

link:../adr/0002-use-git-backed-governed-event-store-for-certificate-profile-lifecycle.md[ADR 0002] chooses Git as a governed, low-throughput event store/source for certificate profile registry/lifecycle events and immutable profile artifact/manifest publication. The rationale is Git's Merkle DAG and hash-linked history, which can provide Bitcoin-like tamper-evidence properties when protected refs, signatures, and operational governance are enforced.

This concept is intentionally bounded:

* Git is not the runtime certificate issuance database.
* Git is not a high-volume event store.
* Git is not a consensus ledger.
* Git is not a query engine.
* No secrets may be stored in Git.

Required stream and publication controls are protected refs with no force-push, signed commits or tags for published/activatable profile states, expected-head / compare-and-swap writes for concurrency, one aggregate stream per ref/path or an equivalent deterministic ordering rule, and no uncontrolled merges in event streams.

Profile lifecycle events are planned to use a structured event envelope containing `eventId`, `aggregateId`, `aggregateVersion`, `eventType`, `occurredAt`, `actor`, `causationId`, `correlationId`, `schemaVersion`, and `payload`. `features/x509/certificate-profile-lifecycle.feature` maps this requirement for `UC-X509-PROFILE`. No lifecycle-envelope domain or runtime behavior is currently implemented.

CQRS read models must be idempotent and replayable. Simple Git-derived reads or caches may be sufficient first; PostgreSQL, OpenSearch, cache projections, or a dedicated event store remain later options if query complexity, latency, or operational scale requires them. The Git adapter belongs to infrastructure behind ports so it can be replaced without changing domain behavior.

Anti-fake status: no implemented Git event-store adapter, event streams, remaining event-envelope fields, projectors, read models, CAS write path, signed-commit publication workflow, replay validation, or standard database/search/cache projections exist yet.

== 8.10 Stable Algorithm Identity and Protocol Taxonomies

Status: confirmed decision / partially implementation-informed.

link:../adr/0003-use-enriched-enums-for-stable-algorithm-wire-identities.md[ADR 0003] separates intrinsic algorithm identities by protocol role. X.509 subject SPKI profiles, X.509 issuer signature suites, provider mechanisms, CVC algorithms, SSH algorithms, and content hashes are distinct closed taxonomies; no implicit cross-taxonomy conversion is permitted. Exact, immutable, case-sensitive `wireToken` values are the serialized/persisted identity. Enum ordinals and Java constant names are never wire or storage identities.

Only stable intrinsic facts belong in these types. Deployment permission, standards edition, transition dates, jurisdiction, provider/module state, certification, and compliance evidence are mutable policy/evidence facts and are excluded. CertificateProfile v1 retains the field `publicKeyAlgorithms` and tokens `EC_P256` and `RSA_3072`; the planned Java rename to `SubjectPublicKeyProfile` must not change its golden canonical bytes, digest, or profile ID.

Anti-fake status: replacement episode 0004 verifies only that the current bare `PublicKeyAlgorithm` enum has exactly `EC_P256` and `RSA_3072`, case-sensitively and without declaration-order semantics. It does not provide explicit `wireToken` data, serialization/parsing, the rename, separate role taxonomies, enriched metadata, or policy admission. Requirements `REQ-X509-SUBJECT-PUBLIC-KEY-PROFILE-001` through `005` remain `@not-implemented`.

== 8.11 Governed Deployment Policy and Evidence

Status: confirmed decision / not implemented.

link:../adr/0004-use-governed-content-addressed-yaml-deployment-policy-snapshots.md[ADR 0004] places mutable deployment eligibility in immutable, content-addressed YAML snapshots outside CertificateProfile v1 identity. Policy is strictly admitted declarative data: it selects exact known wire tokens and invokes only stable rule IDs from a compatible closed catalog of compiled Java validators with typed bounded parameters. Expressions, scripts, reflection/class or method targets, regular expressions, and network references are forbidden.

Evaluation is deterministic and fail-closed. Inputs include an explicit evaluation instant, identified policy and evidence snapshots, operation context, evaluator version, and validator-catalog version; evaluators use no implicit clock or network. Missing, stale, revoked, incompatible, ambiguous, inapplicable, or unverifiable inputs deny the operation with stable reasons. Audit evidence binds the CertificateProfile ID, policy snapshot ID, all evidence IDs, evaluator/catalog versions, explicit instant, result, and stable reason/rule IDs without implying certification.

Publication and reload follow governed signed manifests, trusted distribution, compatibility checks, persistent monotonic rollback state, and `stage -> validate -> atomically swap`. Candidate failure retains the prior complete known-good set; mixed active sets are forbidden. Git remains the governed authoring/history/publication channel, not the runtime policy database, trust anchor, freshness proof, or online decision store.

Anti-fake status: no policy schema, canonical identity, parser, signer, trusted distribution, validator catalog, evidence model, evaluator, audit binding, rollback mechanism, or atomic reload exists. `REQ-X509-DEPLOYMENT-POLICY-001` through `012` are `@not-implemented`; `013` through `016` are decision placeholders.

== 8.12 Testing, Dataset, and Documentation Conventions

Status: confirmed with implementation-informed tooling caveats.

* Gherkin feature files are the executable plan.
* Multiple validation modes must reuse the same feature text.
* Adapter behavior is validated through Cucumber.
* Java domain/application behavior follows JavaSpec red-green-refactor discipline.
* JavaSpec 1.0.0-RC1 can express current record/enum examples, but its observed generation model cannot coherently generate heterogeneous top-level sealed permittees. Such a case requires a `FRAMEWORK_INCOHERENCE` typed stop owned by JavaSpec, not a production-model workaround.
* Passing tests and captured traces are not automatically reinforcement-learning evidence. Episodes 0001–0004 are accepted raw records and curated SFT candidates, with evaluation eligibility marked `NOT_ELIGIBLE`; none is claimed RL-ready. RL readiness requires a separately approved reward contract, state/action evidence, partition controls, hidden verification, and required preference alternatives.
* Generated project artifacts are written in English.
