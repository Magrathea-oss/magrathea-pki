= 10. Quality Requirements

Status: confirmed priorities with planned and implementation-informed validation scenarios.

== 10.1 Quality Tree

[cols="1,1,3",options="header"]
|===
| Priority | Quality | Meaning for Magrathea PKI

| 1
| Security
| Protect keys, trust decisions, certificate lifecycle operations, custody profiles, and administrative access.

| 2
| Correctness
| Enforce PKI protocol rules, key usage rules, policy decisions, and lifecycle transitions accurately.

| 3
| Auditability
| Produce reliable evidence for security, compliance, and operational review.

| 4
| Modularity
| Keep X.509, CVC, SSH, custody, policy, and audit concerns independently evolvable.

| 5
| Testability
| Validate behavior through BDD-first executable specifications and focused Java behavior tests.

| 6
| Operability
| Support reliable operation, diagnostics, and deployment across supported trust profiles.
|===

== 10.2 Quality Scenarios

[cols="1,1,1,3",options="header"]
|===
| Quality | Scenario | Status | Source / Notes

| Security
| A key without `SIGN` usage cannot sign a digest.
| planned
| `features/kms/key-signing.feature`, `@not-implemented`.

| Security
| A key policy requiring HSM denies use when only software custody is available.
| planned
| `features/kms/key-policy.feature`, `@not-implemented`.

| Correctness
| Decryption with a different key than the encryption key fails.
| planned
| `features/kms/key-encryption.feature`, `@not-implemented`.

| Correctness
| Destroyed key handles become invalid for future operations.
| planned
| `features/kms/key-lifecycle.feature`, `@not-implemented`.

| Auditability
| Key creation, signing, destruction, and policy denial create audit events.
| planned
| `features/kms/key-audit.feature`, `@not-implemented`.

| Auditability
| Audit events are append-only and cannot be modified or deleted.
| planned
| `features/kms/key-audit.feature`, `@not-implemented`.

| Correctness / Reproducibility
| The approved six-field TLS baseline produces the exact ADR 0001 canonical JSON bytes, SHA-256 digest `d40ed2a8c1f61164e3f540e7623608c9a8797a7ad1a657d4dffbb661cd1999ee`, and corresponding namespaced ID on every conforming implementation.
| planned / not implemented
| `REQ-X509-PROFILE-INTEGRITY-001`; exact golden-vector acceptance criterion in ADR 0001.

| Correctness / Reproducibility
| Semantically equivalent admitted profiles produce identical IDs despite permitted YAML presentation changes, absent versus explicit optional defaults, unordered-set member order, or canonically equivalent Unicode subject templates.
| planned / not implemented
| `REQ-X509-PROFILE-INTEGRITY-003`, `006`, `007`, and `012`.

| Correctness / Integrity
| Every admitted semantic change that is identity-significant, including changed validity or accepted subject-template case/whitespace, changes the computed ID and fails comparison with the former trusted ID.
| planned / not implemented
| `REQ-X509-PROFILE-INTEGRITY-002` and `013`.

| Security / Correctness
| Each forbidden or ambiguous input class is rejected before any model, canonical JSON, digest, or ID is produced: unknown fields, aliases/anchors, explicit nulls, duplicate keys, duplicate set members, and unknown or miscased enum tokens.
| planned / not implemented
| `REQ-X509-PROFILE-INTEGRITY-004`, `005`, and `008` through `011`.

| Security / Correctness
| Every public expected-ID construction path rejects a value unless the complete untrimmed value has the v1 prefix and exactly 64 lowercase hexadecimal digest characters.
| planned / not implemented
| `REQ-X509-PROFILE-INTEGRITY-014`. Replacement episode 0001 proves value preservation only, not grammar enforcement.

| Security / Auditability
| Before issuance-policy validation, a computed-ID mismatch rejects issuance eligibility and records a `PROFILE_INTEGRITY_FAILED` event with profile path, expected ID, computed ID, actor or system identity, timestamp, and reason.
| planned / not implemented
| `REQ-X509-PROFILE-INTEGRITY-002`; ADR 0001. Identity success must not bypass separate fail-closed issuance-policy validation.

| Modularity
| X.509, CVC, and SSH maintain separate protocol-specific models.
| confirmed architectural constraint
| Chapter 1 and C4 container model.

| Testability
| Feature text is reused across validation modes instead of duplicated.
| confirmed process constraint
| Repository policy.

| Testability
| Full Maven reactor packaging succeeds on the Java 21 / Spring Boot 4.1.0 baseline.
| implementation-informed
| `/tmp/apache-maven-3.9.11/bin/mvn -DskipTests clean package` completed with `BUILD SUCCESS`; all 7 modules succeeded.

| Operability
| Platform dependency versions are managed by Spring Boot rather than explicit Reactor/R2DBC overrides.
| implementation-informed
| Explicit Reactor BOM and `r2dbc-h2` version overrides were removed during the Spring Boot 4 conversion.

| Operability
| Custody backend health and audit failures are observable.
| needs-review
| No executable scenario or operational design yet.
