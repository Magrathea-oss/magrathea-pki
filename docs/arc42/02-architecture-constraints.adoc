= 2. Architecture Constraints

Status: implementation-informed, with confirmed product constraints and the Spring Boot 4 scaffold baseline.

== 2.1 Technical Constraints

[cols="1,1,3",options="header"]
|===
| Constraint | Status | Consequence

| Java 21 target runtime
| confirmed
| Production code and build configuration must remain compatible with Java 21.

| Spring Boot 4 LTS target with Spring Reactive/WebFlux
| confirmed/implementation-informed
| Adapter and bootstrap work use the Spring Boot 4.1.0 reactive stack on Java 21. Spring Boot manages Reactor and related platform dependency versions.

| DDD bounded contexts
| confirmed
| X.509 CA, CVC CA, SSH CA, KMS, HSM Key Custody, and TPM Attestation are modeled as separate bounded contexts.

| Protocol-specific PKI models
| confirmed
| X.509, CVC, and SSH certificate concepts must not be collapsed into one generic certificate model.

| Shared kernels only for truly shared capabilities
| confirmed
| Shared PKI Kernel, Crypto/Key Custody Kernel, Policy Kernel, and Audit Kernel may share language and behavior, but they are not C4 containers and must not erase protocol boundaries.

| HSM and TPM boundary rule
| confirmed
| Product-owned or product-managed HSM and TPM capabilities are inside the Magrathea PKI solution boundary. Physical placement is documented in the Deployment View, not in the system context.

| Compliance targets
| confirmed
| Architecture and implementation decisions must consider FIPS, eIDAS, ETSI, and Common Criteria expectations.

| CertificateProfile v1 identity contract
| confirmed / not implemented
| Every v1 identity implementation must follow the normative six-field admission, normalization, RFC 8785 JCS, SHA-256, and whole-string identifier contract in link:../adr/0001-use-content-addressed-yaml-for-certificate-profiles.md[ADR 0001]. Identity-affecting changes require a new namespace; identity formation must remain separate from fail-closed issuance-policy validation.

| Git-backed governed profile event store guardrails
| confirmed design / not implemented
| link:../adr/0002-use-git-backed-governed-event-store-for-certificate-profile-lifecycle.md[ADR 0002] permits Git only as a governed, low-throughput source for certificate profile registry/lifecycle events and immutable artifact/manifest publication. It forbids secrets in Git, force-pushed event streams, uncontrolled stream merges, and use of Git as a runtime issuance database, high-volume event store, consensus ledger, or query engine. Git integration must remain replaceable infrastructure behind ports.

| Protocol-specific algorithm taxonomies and stable wire identities
| confirmed design / partially implementation-informed
| link:../adr/0003-use-enriched-enums-for-stable-algorithm-wire-identities.md[ADR 0003] requires separate closed taxonomies for X.509 subject SPKI profiles, X.509 issuer signature suites, provider mechanisms, CVC, SSH, and content hashes. Serialized identity must use explicit immutable case-sensitive wire tokens, never enum ordinals or Java names. The current bare `PublicKeyAlgorithm` vocabulary is only `EC_P256` and `RSA_3072`; enriched metadata, protocol-role types, parsing, and the planned `SubjectPublicKeyProfile` rename are not implemented.

| Governed deployment-policy snapshots
| confirmed design / not implemented
| link:../adr/0004-use-governed-content-addressed-yaml-deployment-policy-snapshots.md[ADR 0004] requires strictly admitted, signed, immutable, content-addressed YAML snapshots that select known wire tokens and invoke only compatible compiled Java validators. Evaluation must be deterministic at an explicit instant, local and network-free, fail closed, bind immutable evidence, reload atomically, resist rollback, and remain outside CertificateProfile v1 identity. YAML must not define algorithms or executable behavior.
|===

== 2.2 Organizational and Process Constraints

[cols="1,1,3",options="header"]
|===
| Constraint | Status | Consequence

| BDD-first development
| confirmed
| Gherkin feature files are the executable plan and the source of truth for requirements.

| Red test before production behavior
| confirmed
| At least one meaningful RED test must exist before production code changes.

| Multi-runner rule
| confirmed
| Feature text is not duplicated for different validation modes; runners, glue, profiles, or adapters define validation differences.

| Thin adapter rule
| confirmed
| Adapters extract request data, delegate, and form responses. They must not contain domain rules, orchestration, persistence logic, cryptographic decisions, or production state.

| Anti-fake completion rule
| confirmed
| Support status must be explicit. Current complete behavior scenarios are tagged `@not-implemented`; deployment-policy requirements `013` through `016` are explicitly `@placeholder`. Neither status may be reported as supported behavior, and accepted domain slices do not by themselves complete a feature scenario.

| English generated artifacts
| confirmed
| Documentation, ADRs, source comments, and user-facing generated text are written in English.
|===

== 2.3 Architecture Modeling Constraints

[cols="1,1,3",options="header"]
|===
| Constraint | Status | Consequence

| C4 containers are bounded contexts
| confirmed
| Containers represent deployable, independently releasable, or intentionally extractable bounded contexts. Maven modules, JARs, WARs, Docker images, and shared kernels are not C4 containers.

| C4 diagrams owned by c4model
| confirmed
| ARC42 integrates `docs/c4/workspace.dsl`; diagram creation or structural changes are handled by the c4model agent.

| Deployment details separated from C2
| confirmed
| Packaging artifacts and physical HSM/TPM placement belong to Chapter 7, not the C2 container model.
|===

== 2.4 Current Scaffold Constraints and Gaps

[cols="1,1,3",options="header"]
|===
| Item | Status | Constraint or Gap

| Current Maven scaffold
| implementation-informed
| Modules exist for domain, application, infrastructure, API adapter, CLI adapter, and bootstrap application.

| Spring Boot platform baseline
| confirmed/implementation-informed
| Root `pom.xml` uses Spring Boot 4.1.0 with Java 21. The explicit Reactor BOM override was removed so Boot manages Reactor; explicit `r2dbc-h2` version management was removed and runtime scope is set where needed. Cucumber is 7.34.4 and Picocli is 4.7.7.

| Missing intended modules
| implementation-informed
| `trust-engine-admin-adapter` and `trust-engine-test-support` are expected by repository policy but are not present yet.

| Feature coverage
| implementation-informed
| Existing feature files cover X.509 profile integrity, profile lifecycle event-envelope metadata, subject-public-key profile taxonomy, deployment-policy snapshots, and KMS. Broader X.509 profile management beyond the envelope metadata, issuance, revocation, validation, CVC, SSH, HSM custody, and TPM attestation requirements are not yet represented as executable features.

| Implementation depth
| implementation-informed
| The bootstrap entry point exists. Accepted replacement episodes 0001–0003 prove that `CertificateProfileId` preserves a canonical supplied value and rejects leading whitespace and explicit null under the complete v1 grammar. Episode 0004 proves only that the bare `PublicKeyAlgorithm` enum has exactly the case-sensitive tokens `EC_P256` and `RSA_3072`, without declaration-order semantics. No complete feature scenario is implemented: enriched wire-token metadata, exact token parsing, the `SubjectPublicKeyProfile` rename, separate role taxonomies, policy YAML/schema/canonical identity, validator catalog/evaluator, evidence, publication, reload, and all remaining profile identity/application behavior are absent.
|===
