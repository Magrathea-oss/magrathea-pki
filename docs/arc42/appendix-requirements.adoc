= Appendix: Requirements

Status: implementation-informed from current Gherkin under `features/x509` and `features/kms`, with anti-fake implementation status.

All observed feature scenarios are tagged `@not-implemented`. After the dataset reset, the only accepted X.509 pure-domain implementation evidence is replacement episode 0001: `CertificateProfileId` preserves and exposes its supplied `String value`. This does not enforce the normative v1 grammar and does not satisfy any complete feature scenario. The discarded 33-behavior corpus is not current evidence.

== A.1 Documentation Ownership and Status

This appendix owns the executable use-case catalogue, PlantUML use-case overview, use-case-to-Gherkin mapping, and scenario catalogue. Chapter 3 owns system context instead.

[cols="1,3",options="header"]
|===
| Status | Meaning

| planned
| Roadmap intent exists, but executable scenarios or implementation are incomplete.

| implementation-informed
| Backed by current repository evidence such as Gherkin, source, specifications, or accepted dataset evidence.

| not implemented
| Must not be reported as supported behavior; this corresponds to `@not-implemented`.

| needs-review
| Requires human review or architectural/requirements clarification.
|===

Use cases, orchestration, Cucumber validation, and adapters belong to `story-driven`. Pure domain behavior belongs to `spec-driven`; a domain slice does not replace use-case validation. Application-use-case Javadocs should link to the stable use-case anchor and mapped feature file.

=== A.6.1 Certificate Profile Integrity

Source: `features/x509/certificate-profile-integrity.feature`.

Normative architecture source: link:../adr/0001-use-content-addressed-yaml-for-certificate-profiles.md[ADR 0001]. The feature exercises the six-field identity contract at requirement level: strict YAML/schema admission, default expansion, NFC subject normalization, sorting only declared unordered sets, RFC 8785 JCS bytes, SHA-256 identity, complete ID grammar, trusted-reference comparison, and mismatch audit evidence. Identity formation is not issuance-policy approval.

=== A.6.2 Certificate Profile Lifecycle

Source: `features/x509/certificate-profile-lifecycle.feature`.

[cols="2,1,1,1,2",options="header"]
|===
| Scenario | Requirement ID | Use Case | Classification | Declared / Validated Status

| Register a certificate profile lifecycle change as a structured event envelope
| `REQ-X509-PROFILE-LIFECYCLE-001`
| <<UC-X509-PROFILE,UC-X509-PROFILE>>
| functional
| `@not-implemented`; no lifecycle-envelope domain or use-case behavior exists
|===

The requirement covers a structured envelope with `eventId`, `aggregateId`, `aggregateVersion`, `eventType`, `occurredAt`, `actor`, `causationId`, `correlationId`, `schemaVersion`, and `payload`, without secrets. Persistence, stream layout, CAS writes, signed publication, projections, replay, and runtime Git integration are explicitly outside this scenario and remain not implemented.

== A.7 KMS Scenario Catalogue

All entries below are declared functional requirements and remain `@not-implemented`.

| <<UC-CUSTODY-CONFIG,UC-CUSTODY-CONFIG>>

| <<UC-AUDIT-REVIEW,UC-AUDIT-REVIEW>>
| Key creation generates KEY_CREATED event; Key signing generates KEY_USED_FOR_SIGN event; Key destruction generates KEY_DESTROYED event; Policy denial generates POLICY_DENIED_OPERATION event; Audit events are append-only and immutable.

| actionable
| Implement CertificateProfile v1 incrementally against `REQ-X509-PROFILE-INTEGRITY-001` through `014`. Current evidence stops at identifier-value preservation; grammar enforcement and the complete identity/integrity mechanism are absent.

| actionable
| Implement `REQ-X509-PROFILE-LIFECYCLE-001` before claiming lifecycle-envelope support. Define additional lifecycle requirements separately from persistence decisions.

| actionable
| Add executable coverage for `UC-X509-ISSUE`, `UC-X509-REVOKE`, and `UC-X509-VALIDATE`; identity success must not substitute for issuance-policy requirements.

| Add CVC and SSH requirements when those roadmap phases begin.
