= Appendix: Requirements

Status: implementation-informed from current Gherkin under `features/x509` and `features/kms`, with anti-fake implementation status.

Current complete behavior scenarios are tagged `@not-implemented`; deployment-policy requirements `REQ-X509-DEPLOYMENT-POLICY-013` through `016` are explicitly `@placeholder` because prerequisite decisions are unresolved. Accepted replacement episodes 0001–0003 prove only `CertificateProfileId` canonical-value preservation, leading-whitespace rejection, and explicit-null rejection under its complete v1 grammar. Episode 0004 proves only the exact bare `PublicKeyAlgorithm` vocabulary `EC_P256` and `RSA_3072`, without declaration-order semantics. No accepted slice satisfies a complete feature scenario. The discarded historical 33-behavior corpus is not current evidence.

== A.1 Documentation Ownership and Status

This appendix owns the executable use-case catalogue, PlantUML use-case overview, use-case-to-Gherkin mapping, and scenario catalogue. Chapter 3 owns system context instead.

[cols="1,3",options="header"]
|===
| Status | Meaning

| planned
| Roadmap intent exists, but executable scenarios or implementation are incomplete.

| implementation-informed
| Backed by current repository evidence such as Gherkin, source, specifications, or accepted dataset evidence.

| not implemented
| Must not be reported as supported behavior; this corresponds to `@not-implemented`.

| placeholder
| Records an unresolved prerequisite decision and future executable acceptance criteria; this corresponds to `@placeholder` and is not support.

| needs-review
| Requires human review or architectural/requirements clarification.
|===

Use cases, orchestration, Cucumber validation, and adapters belong to `story-driven`. Pure domain behavior belongs to `spec-driven`; a domain slice does not replace use-case validation. Application-use-case Javadocs should link to the stable use-case anchor and mapped feature file.
